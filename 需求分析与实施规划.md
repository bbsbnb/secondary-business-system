# 天行建筑智能管理平台 — 需求分析与实施规划

> 基于《全流程开发手册 v1.11》深度解析 | 2026年7月

## 一、系统全景概览

### 1.1 业务定位
建筑施工企业**月度经营全流程管理系统**，覆盖从一次经营交底到月度复盘的完整业务闭环。

### 1.2 核心数据流

```
一次经营(M1基线) → 任务分解(M2) → 策划编制审核(M3/M4) → 回款落实(M5)
    ↓
认质认价(M6) → 联系单(M7) → 签证(M8) → 索赔(M9) → 变更(M10)
    ↓
月验工计价(M11) → 材料结算(M12) → 消耗核定(M13) → 9项检查(M14-M23)
    ↓
建造合同3表关联(M24) → 月度复盘(M25) → 下月M1
```

### 1.3 版本策略（v1.11关键变更）

| 轨道 | 内容 | 说明 |
|------|------|------|
| 🚀 **轨道A — 审批流骨架** | 占一期80%工作量 | 不依赖AI，第1周即可启动 |
| 🧠 **轨道B — AI能力** | 7项AI场景 | 依赖OCR验证，OCR达标→继续；不达标→手写签证人工录入 |

### 1.4 已确认不做/取消

- ❌ 外部H5审批 → 改线下签字+上传
- ❌ 企微小程序 → 改飞书自建应用
- ❌ 库存扣减联动 → 暂不实现

## 二、16个核心节点逐层解析

### M1：上月复盘归档 + 一次经营数据录入 `一期·基础`

**业务本质**：项目启动基线锁定，不可逆

| 维度 | 内容 |
|------|------|
| 责任人 | 项目经理主持；经营专员/造价员/合同资料员/技术员录入 |
| 核心动作 | 归集一次经营交底资料+上月复盘资料统一归档 |
| 6项基线数据 | ①中标合同价 ②标后成本(18张附表) ③利润留存点(≥8%) ④不平衡报价策略 ⑤合同关键条款 ⑥总进度计划 |
| 审批模式 | 人工录入 → 确认人复核 → 系统标记"基线已锁定" |
| 铁律 | **基线锁定不可逆，修改需申请解锁+留痕** |
| 附件 | 施工合同扫描件、招投标文件、建造合同、施组设计、交底签到表 |

**关键逻辑**：
- 一次经营（投标→合同签订→目标成本锁定）**不在系统内运行审批流程**
- 仅在M-1时人工录入6项基线数据，作为后续所有二次经营的基准
- 基线一旦锁定，任何修改必须走解锁流程并留痕审计

### M2：本月任务目标分解 `一期·审批流`

**业务本质**：将总计划分解为月计划

| 维度 | 内容 |

| 责任人 | 技术总工编制 → 项目经理审核 → 分管领导审批 |

**注意**：审批流程说明中有矛盾——正文写"技术总工编制"，mermaid图写"项目经理编制"。根据开发手册最终确认：**生产经理编制上传→项目经理→工程部→生产副总→资料室归档**。

## 三、审批引擎核心架构设计

### 3.1 六层抽象模型

```
L1: 流程定义层 (ApprovalStepTemplate)
    └─ 每个节点预定义审批步骤模板
L2: 实例创建层 (ApprovalInstance)
    └─ 根据模板创建具体审批实例
L3: 步骤推进层 (ApprovalStep)
    └─ 当前审批步骤的状态管理
L4: 条件分支层 (ConditionRouter)
    └─ 按金额/类型自动路由
L5: 并行审核层 (ParallelReviewGroup)
    └─ 多人同时审核，全部通过才流转
L6: 审计追踪层 (AuditLog)
    └─ 所有操作留痕
```

### 3.2 四种审批模式

| 模式 | 适用节点 | 特点 |
|------|---------|------|
| **串行审批** | M2/M4/M6/M7/M8/M9/M10/M12/M13/M25 | 上一步完成才流转下一步 |
| **并行审核** | M11 | 6人同时审核，全部通过才流转 |
| **条件分支** | M8(金额)/M9(金额) | 按条件自动路由不同审批人 |
| **自查上传** | M14-M23 | 无审批链，上传+整改闭环 |

### 3.3 铁律约束体系

| 约束码 | 含义 | 影响范围 | 实现方式 |
|--------|------|---------|---------|
| `T1` | 总经理终审 | M8/M9/M11/M12 | 审批链最后一步强制总经理 |
| `T2` | 造价部强制前置 | M4/M6/M7/M8/M9/M10/M11/M12 | 造价部节点在所有金额相关节点之前 |
| `T3` | 法代联签(>50万) | M8 | 金额>50万自动添加法务代表节点 |
| `T5` | 经营副总牵头 | M4 | 经营副总作为终审节点 |
| `T6` | 超概说明(3日) | M11/M24 | 累计达100%触发，3日未说明自动升级预警 |

### 3.4 审批状态机

```
draft(草稿) → submitted(已提交) → active(审批中) → completed(已完成)
                                        ↘ rejected(已驳回)
                                        ↘ returned(已退回)
```

## 八、关键设计决策记录

### 决策1：M1基线锁定不可逆
- **原因**：基线是后续所有二次经营的依据，必须保证数据一致性
- **实现**：锁定后修改需走解锁流程+留痕审计

### 决策2：M3和M4共用同一业务实体
- **原因**：M3是编制，M4是审核，属于同一份策划书的不同阶段
- **实现**：通过 `business_form_id` 关联，M4审核通过后才算定稿

### 决策3：M11采用平行审核模式
- **原因**：施工员/材料员/质量员/安全员/生产经理/项目经理需同时审核
- **实现**：ParallelReviewGroup收集6人意见，全部通过才流转下一步

### 决策4：T6超概自动升级预警
- **原因**：防止超概被忽视，确保管理层及时介入
- **实现**：累计达100%触发预警，3日未提交书面说明自动升级至经营副总+总经理

### 决策5：外部签章采用线下签字+上传凭证
- **原因**：v1.11已确认不做外部H5审批
- **实现**：甲方/监理签章线下完成后，拍照/扫描上传系统

### 决策6：M9双28天精确计算工作日
- **原因**：超时=丧失索赔权，法律意义重大
- **实现**：使用工作日计算(排除节假日)，设置倒计时提醒

## 九、风险与应对

| 风险 | 影响 | 应对方案 |
|------|------|---------|
| OCR识别准确率不足 | AI能力无法推进 | 手写签证人工录入，AI只审打印件 |
| 多项目数据隔离遗漏 | 数据泄露 | 所有查询强制 `WHERE project_id`，代码审查重点 |
| 审批引擎并发冲突 | 数据不一致 | 乐观锁 + 数据库事务隔离 |
| 飞书API限流 | 通知延迟 | 消息队列缓冲 + 降级为站内通知 |
| 大附件上传超时 | 用户体验差 | MinIO分片上传 + 进度条显示 |
